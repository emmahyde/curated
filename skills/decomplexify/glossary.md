# Glossary — Code Complete (2nd ed.)

Key terms with one-line definitions. `(Ch N)` points to the chapter where the term is developed.

- **Abstract Data Type (ADT)** — a type defined by its operations and semantics, independent of implementation; every class implements one (Ch 6)
- **Abstraction** — a model that ignores irrelevant detail; the principal tool for conquering complexity (Ch 5, Ch 34)
- **Accidental difficulty** — difficulty arising from chosen tools/process, not from the problem itself (Ch 5, Ch 30)
- **Access routine** — getter/setter mediating all reads/writes to data, enabling validation and instrumentation (Ch 13)
- **Assertion** — a runtime check documenting a condition that must always be true; for bugs, not expected errors (Ch 8)
- **Barricade** — an architectural boundary that sanitizes all external data on entry: dirty outside, clean inside (Ch 8)
- **Binding time** — the lifecycle point when a variable's value becomes fixed (compile/load/creation/execution); earlier is faster, later is more flexible (Ch 10)
- **Boundary condition** — the edge of an input range, where errors cluster; test min, min+1, max−1, max (Ch 12, Ch 22)
- **Brooks's Law** — adding people to a late software project makes it later (Ch 28)
- **Cardinal Rule of Software Evolution** — internal quality must improve, or at least not degrade, with every change (Ch 24)
- **Code smell** — a surface indicator (duplication, long routine, large parameter list, feature envy) signaling deeper structural debt (Ch 24)
- **Code tuning** — source-level changes to a working, correct program to reduce time or memory; the inverse of refactoring (Ch 25, Ch 26)
- **Cohesion** — how strongly the operations within a module relate; functional cohesion (one operation) is the ideal (Ch 6, Ch 7)
- **Configuration management (SCM)** — systematically tracking artifacts and changes so the system keeps its integrity over time (Ch 28)
- **Conceptual integrity** — a single coherent vision reflected top-to-bottom through architecture and code (Ch 3)
- **Construction** — the central hands-on activity: detailed design, coding, debugging, unit/integration testing, integration (Ch 1)
- **Correctness vs. robustness** — competing error philosophies: never return a wrong result vs. always keep running (Ch 8, Ch 20)
- **Coupling** — the degree of dependence between two modules; keep it small, visible, and flexible (Ch 5, Ch 7)
- **Cyclomatic complexity (McCabe)** — count of decision points in a routine; 0–5 fine, 6–10 simplify, 10+ decompose (Ch 19)
- **Dangling pointer** — a freed pointer not set to null; use-after-free yields unpredictable behavior (Ch 13)
- **Defect cost escalation** — a requirements defect costs ~1× to fix at requirements time, 10–100× after release (Ch 3)
- **Defensive programming** — writing code that protects itself from invalid input and other programmers' mistakes (Ch 8)
- **Design heuristics** — an ordered toolkit (find objects, form abstractions, encapsulate, hide secrets, identify likely changes, loosen coupling, use patterns) (Ch 5)
- **Diseconomy of scale** — software has no economies of scale; bigger projects cost more per unit and have higher defect density (Ch 27)
- **DRY (Don't Repeat Yourself)** — duplication is a design error; eliminate copy-and-paste (Ch 24)
- **Egoless programming** — separating personal identity from code so criticism can be accepted; code is team property (Ch 21, Ch 33)
- **Encapsulation** — the enforcer of abstraction; prevents access to internals (Ch 5, Ch 6)
- **Enumerated type** — a type whose legal values are a named finite set; reserve the first entry as invalid (Ch 12)
- **Equivalence partitioning** — one representative test per class of inputs expected to behave identically (Ch 22)
- **Fundamental Theorem of Formatting** — good layout shows the logical structure of the program (Ch 31)
- **General Principle of Software Quality** — improving quality reduces development cost by eliminating rework (Ch 20)
- **Guard clause** — an early return at routine entry that handles degenerate input before the main logic (Ch 17)
- **Heuristic** — exploratory, suggestive guidance; a searchlight, not a road map (contrast: algorithm) (Ch 2)
- **Information hiding** — designing interfaces by asking "what must this hide?"; hides volatile, likely-to-change details (Ch 5)
- **Intellectual honesty** — accurate self-assessment and truthful communication without defensiveness (Ch 33)
- **Intent comment** — a comment explaining *why* at the problem-domain level; the most valuable comment type (Ch 32)
- **Incremental integration** — adding one component at a time so failures are always traceable to the newest addition (Ch 29)
- **Law of Demeter** — only call routines on your own class, objects you create, are passed, or contain (Ch 6)
- **Live time** — total statements from a variable's first to last reference; a measure of tracking burden (Ch 10)
- **Loop-with-exit** — a one-entry/one-exit loop with the exit condition gathered mid-body; aids comprehension (Ch 16)
- **Magic number** — a literal value embedded without explanation; replace with a named constant (Ch 12)
- **Managing complexity** — Software's Primary Technical Imperative; the meta-criterion for construction decisions (Ch 5, Ch 34)
- **Metaphor (software)** — a way of looking at development; construction > farming > writing for general use (Ch 2)
- **Pair programming** — two programmers at one keyboard (driver + navigator); detection rate near formal inspection (Ch 21)
- **Pareto Principle (80/20)** — ~20% of code uses ~80% of run time; profiling reveals which 20% (Ch 25)
- **Premature optimization** — tuning before the program is correct and before measurement (Ch 25)
- **Principle of Proximity** — when no ordering constraint exists, keep related statements close together (Ch 10, Ch 14)
- **Problem definition** — a 1–2 page statement of the problem, in problem terms, with no solution references (Ch 3)
- **Profiling** — instrumenting a running program to locate the actual hot spots (Ch 25, Ch 26)
- **Programming into a language** — deciding what you want, then expressing it with the language's primitives (vs. *in* the language, letting limits dictate design) (Ch 4, Ch 34)
- **Pseudocode Programming Process (PPP)** — build a routine by writing intent-level pseudocode, then translating it line-by-line into code (Ch 9)
- **Psychological distance** — difficulty seeing errors in one's own code due to a shared mental model (Ch 23, Ch 32)
- **Refactoring** — changing internal structure without changing observable behavior (Fowler) (Ch 24)
- **Regression testing** — re-running prior tests after changes to confirm nothing broke; the safety net for refactoring (Ch 22, Ch 24)
- **Reserve parachute** — memory allocated at startup and freed on out-of-memory to allow graceful shutdown (Ch 13)
- **Scientific method of debugging** — stabilize → locate → hypothesize → experiment → prove → fix → test → look for similar defects (Ch 23)
- **Self-documenting code** — code so clear that structure and naming carry most of the explanatory load (Ch 32)
- **Smoke test** — an end-to-end check verifying a build is stable enough for further testing (Ch 29)
- **Span** — average lines between consecutive references to a variable; minimize toward zero (Ch 10)
- **Strength reduction** — replacing an expensive operation with a cheaper equivalent (Ch 26)
- **Structured basis testing** — minimum path-coverage test count = 1 + one per decision keyword (Ch 22)
- **Structured programming** — building programs only from sequence, selection, and iteration, each one-entry/one-exit (Ch 17, Ch 19)
- **Table-driven method** — replacing procedural logic with a data structure mapping inputs to outputs (direct/indexed/stair-step access) (Ch 18)
- **Technology wave position** — where a stack sits in its maturity cycle; early-wave (buggy, sparse docs) vs. late-wave (mature) (Ch 4)
- **Test-first development** — write a failing test, write minimum code to pass, refactor, repeat (Ch 9, Ch 22)
- **Wicked problem** — one that can only be clearly defined by (partially) solving it first; design is wicked (Ch 5)
